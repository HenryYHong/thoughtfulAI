Multi-Agent Systems: When One Model Is Not Enough

~8 min read · Multi-Agent Systems · Week 9

Why multi-agent now

This week made one thing clear: many useful AI problems are not just "answer one prompt well." They are workflows. Planning, research, verification, and formatting often fight each other when forced into one monolithic prompt.

Multi-agent systems break that bottleneck by splitting work across specialized roles. Instead of asking one model to do everything at once, we design a coordination loop where different agents contribute focused pieces.

A multi-agent system is less about "more models" and more about "better division of cognitive labor."

Single-agent vs multi-agent

A strong single agent can still be best for short, low-ambiguity tasks. But as tasks become longer and higher-stakes, role specialization starts to help.

  • Single Agent One loop, one policy. Lower overhead, easier debugging, but less explicit decomposition.
  • Planner Agent Builds an execution graph: goals, subgoals, dependencies, and stop conditions.
  • Worker Agents Execute narrow subtasks like retrieval, coding, or critique.
  • Judge Agent Evaluates outputs against rubric constraints before results are accepted.

What changed my thinking is that coordination itself is a design variable. You do not just pick better prompts; you pick topology: who talks to whom, when, and with what memory.

Three useful patterns

The lecture examples map nicely to three architecture patterns I want to experiment with:

  • Planner-Executor-Critic: planner decomposes, executor produces draft outputs, critic catches gaps and requests revisions.
  • Debate + Judge: two agents argue competing solutions, then a separate judge selects or synthesizes a final answer.
  • Router + Specialists: a front agent classifies intent and forwards to domain specialists (math, policy, coding, retrieval).

Design takeaway

Performance is not only a model capability question; it is also an orchestration question. Better routing and critique can outperform brute-force prompting.

Where things break

Multi-agent setups introduce new failure modes that do not show up in single-agent pipelines:

  • Coordination drift: agents optimize local objectives but miss the global task.
  • Error amplification: an early hallucination gets repeated and "validated" downstream.
  • Protocol mismatch: agents use inconsistent formats, causing silent handoff failures.
  • Cost creep: too many loops and retries erase quality-per-dollar gains.

This is where systems thinking matters: we need explicit schemas for messages, bounded retries, and checkpoints that can fail safely instead of cascading.

How I would evaluate it

If I were benchmarking a multi-agent workflow, I would track both task quality and orchestration quality:

  • Outcome quality: correctness, completeness, and user-rated usefulness.
  • Process quality: number of loops, tool errors recovered, and agreement between critic and judge.
  • Robustness: performance under adversarial or ambiguous inputs.
  • Efficiency: latency and token/tool cost per successful outcome.

A system that is slightly less fluent but consistently verifiable and recoverable may be better in production than a flashy system that fails silently.

A small build idea

A project idea from this week: build a "syllabus-to-study-plan" assistant using a router plus two specialists.

  • Router: classify user goal (review notes, timeline planning, project prep).
  • Retrieval specialist: extract relevant week/module context from course docs.
  • Planning specialist: generate a concrete schedule with milestones and checks.
  • Critic: verify deadlines and detect missing prerequisites before returning.

It is small enough to prototype, but complex enough to surface the real coordination issues discussed in class.

What clicked for me

The key shift for me is treating prompts as interfaces, not just instructions. In multi-agent systems, each message is a contract: expected input shape, constraints, and what counts as "done." Once I saw that, the architecture became much more concrete.

I also realized that evaluation needs to include the collaboration process itself. If one agent keeps rescuing another's mistakes, that can hide brittle behavior until deployment.

Open questions

How should we decide when a task deserves multi-agent orchestration versus a better single-agent prompt? Can we learn dynamic routing policies that trade off quality and cost in real time? And what is the right memory strategy so agents share just enough context without overwhelming each other?

Week 9 made me optimistic: multi-agent systems feel like a practical path from "smart model demos" to reliable, auditable AI workflows.
